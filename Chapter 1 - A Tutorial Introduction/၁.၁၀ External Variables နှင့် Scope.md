## 1.10 External Variables နှင့် Scope

main ရှိ ကိန်းရှင်များဖြစ်သော line၊ longest စသည်တို့သည် main အတွက် private သို့မဟုတ် local ဖြစ်သည်။ ၎င်းတို့ကို main အတွင်းတွင် ကြေညာထားသောကြောင့် အခြား function များက ၎င်းတို့သို့ တိုက်ရိုက် ဝင်ရောက်ခွင့် မရနိုင်ပါ။ အခြား functions များရှိ ကိန်းရှင်များအတွက်လည်း တူညီသည်။ ဥပမာ getline ရှi ကိန်းရှင် i သည် copy ရှိ i နှင့် မဆက်စပ်ပါ။ Function တစ်ခုရှိ local ကိန်းရှင်တစ်ခုချင်းစီသည် function ကို ခေါ်သောအခါမှသာ ဖြစ်တည်လာပြီး function ထွက်သောအခါ ပျောက်ကွယ်သည်။ ဤကြောင့် ထိုကဲ့သို့ ကိန်းရှင်များကို များသောအားဖြင့် automatic variables ဟု လူသိများသည်၊ အခြားဘာသာစကားများရှိ ဝေါဟာရများအတိုင်း။ ကျွန်ုပ်တို့ ယခုနောက် ဤ local ကိန်းရှင်များကို ရည်ညွှန်းရန် automatic ဟူသော အသုံးအနှုန်းကို အသုံးပြုမည်။ (Chapter 4 က static storage class ကို ဆွေးနွေးသည်၊ ၎င်းတွင် local ကိန်းရှင်များသည် ခေါ်ဆိုမှုများကြား ၎င်းတို့၏ တန်ဖိုးများကို ထိန်းသိမ်းထားသည်။)

Automatic ကိန်းရှင်များသည် function invocation နှင့်အတူ လာပြီးသွားသောကြောင့် ၎င်းတို့သည် တစ်ခေါ်မှ နောက်တစ်ခေါ်သို့ ၎င်းတို့၏ တန်ဖိုးများကို မထိန်းသိမ်းပါ၊ entry တစ်ခုချင်းစီတွင် အတိအကျ သတ်မှတ်ရမည်။ ၎င်းတို့ကို မသတ်မှတ်ပါက အမှိုက်များ ပါဝင်လိမ့်မည်။

Automatic ကိန်းရှင်များ၏ အခြားရွေးချယ်စရာတစ်ခုအနေဖြင့် functions အားလုံး၏ ပြင်ပတွင် ရှိသော ကိန်းရှင်များ၊ ဆိုလိုသည်မှာ မည်သည့် function မဆို အမည်ဖြင့် ဝင်ရောက်နိုင်သော ကိန်းရှင်များကို သတ်မှတ်နိုင်သည်။ (ဤယန္တရားသည် Fortran COMMON သို့မဟုတ် အပြင်ဆုံး block တွင် ကြေညာထားသော Pascal ကိန်းရှင်များနှင့် တော်တော် တူသည်။) External ကိန်းရှင်များသည် globally ဝင်ရောက်နိုင်သောကြောင့် functions များကြား ဒေတာ ဆက်သွယ်ရန်အတွက် argument lists များ အစား အသုံးပြုနိုင်သည်။ ထို့အပြင် external ကိန်းရှင်များသည် functions များကို ခေါ်ပြီး ထွက်သည့်အခါ ပေါ်လာ၍ ပျောက်ကွယ်ခြင်းထက် အမြဲတမ်း တည်ရှိနေသောကြောင့် ၎င်းတို့ကို သတ်မှတ်ခဲ့သော functions များ ပြန်လာပြီးနောက်တောင် ၎င်းတို့၏ တန်ဖိုးများကို ထိန်းသိမ်းထားသည်။

External ကိန်းရှင်တစ်ခုကို တစ်ကြိမ်တည်းသာ မည်သည့် function ၏ ပြင်ပတွင် အတိအကျ သတ်မှတ်ရမည်။ ဤသည်က ၎င်းအတွက် သိုလှောင်မှုကို ခွဲထုတ်သည်။ ကိန်းရှင်ကို ၎င်းသို့ ဝင်ရောက်လိုသော function တစ်ခုချင်းစီတွင်လည်း ကြေညာရမည်။ ဤသည်က ကိန်းရှင်၏ အမျိုးအစားကို ဖော်ပြသည်။ ကြေညာချက်သည် အတိအကျ extern statement တစ်ခု ဖြစ်နိုင်သည် သို့မဟုတ် အကြောင်းအရာမှ သွယ်ဝိုက်ခြင်း ဖြစ်နိုင်သည်။ ဆွေးနွေးချက်ကို တိကျစေရန် line၊ longest နှင့် max ကို external ကိန်းရှင်များအဖြစ် ထား၍ အရှည်ဆုံး-စာကြောင်း ပရိုဂရမ်ကို ပြန်လည်ရေးကြပါစို့။ ဤသည်က functions သုံးခုလုံး၏ ခေါ်ဆိုမှုများ၊ ကြေညာချက်များနှင့် bodies များကို ပြောင်းလဲရန် လိုအပ်သည်။

```

#include
#define MAXLINE 1000 /* maximum input line size */
int max; /* maximum length seen so far */
char line[MAXLINE]; /* current input line */
char longest[MAXLINE]; /* longest line saved here */
int getline(void);

void copy(void);

/* print longest input line; specialized version */

main()

{
int len;
extern int max;
extern char longest[];
max = 0;
while ((len = getline()) > 0)
	if (len > max) {
		max = len;
		copy();
}

if (max > 0) /* there was a line */
	printf("%s", longest);
return 0;

}

/* getline: specialized version */

int getline(void)
{

int c, i;
extern char line[];

for (i = 0; i < MAXLINE - 1

	&& (c=getchar)) != EOF && c != '\n'; ++i)

	line[i] = c;

if (c == '\n') {

	line[i] = c;

	++i;

}

line[i] = '\0';
return i;

}

/* copy: specialized version */

void copy(void)
{

int i;
extern char line[], longest[];
i = 0;

while ((longest[i] = line[i]) != '\0')
	++i;
}
```

main၊ getline နှင့် copy ရှိ external ကိန်းရှင်များကို အထက်က ဥပမာ၏ ပထမစာကြောင်းများက သတ်မှတ်သည်၊ ၎င်းတို့၏ အမျိုးအစားကို ဖော်ပြပြီး ၎င်းတို့အတွက် သိုလှောင်မှု ခွဲဝေပေးစေသည်။ ဝါကျဖွဲ့ပုံအရ external သတ်မှတ်ချက်များသည် local ကိန်းရှင်များ၏ သတ်မှတ်ချက်များကဲ့သို့ပင် ဖြစ်သည်၊ သို့သော် ၎င်းတို့သည် functions များ၏ ပြင်ပတွင် ဖြစ်ပေါ်သောကြောင့် ကိန်းရှင်များသည် external ဖြစ်သည်။ Function တစ်ခုက external ကိန်းရှင်တစ်ခုကို အသုံးမပြုမီ ကိန်းရှင်၏ အမည်ကို function သို့ သိစေရမည်။ ကြေညာချက်သည် ထပ်ဆောင်း keyword extern မှလွဲ၍ အရင်နှင့် တူညီသည်။

အချို့ အခြေအနေများတွင် extern ကြေညာချက်ကို ချန်လှပ်နိုင်သည်။ External ကိန်းရှင်၏ သတ်မှတ်ချက်သည် သီးခြား function တစ်ခုတွင် ၎င်းကို အသုံးပြုခြင်းမတိုင်မီ source file တွင် ဖြစ်ပေါ်ပါက function တွင် extern ကြေညာချက် မလိုအပ်ပါ။ main၊ getline နှင့် copy ရှိ extern ကြေညာချက်များသည် ထို့ကြောင့် အပိုဆောင်း ဖြစ်သည်။ တကယ်တော့ ဘုံအလေ့အကျင့်မှာ external ကိန်းရှင်များအားလုံး၏ သတ်မှတ်ချက်များကို source file ၏ အစတွင် ထားရှိပြီး extern ကြေညာချက်များအားလုံးကို ချန်လှပ်ခြင်း ဖြစ်သည်။

ပရိုဂရမ်သည် source files များစွာတွင် ရှိပြီး ကိန်းရှင်တစ်ခုကို file1 တွင် သတ်မှတ်ပြီး file2 နှင့် file3 တွင် အသုံးပြုပါက ကိန်းရှင်၏ ပေါ်လာမှုများကို ချိတ်ဆက်ရန် file2 နှင့် file3 တွင် extern ကြေညာချက်များ လိုအပ်သည်။ ပုံမှန်အလေ့အကျင့်မှာ ကိန်းရှင်များနှင့် functions များ၏ extern ကြေညာချက်များကို သမိုင်းအရ header ဟု ခေါ်သော သီးခြား file တစ်ခုတွင် စုဆောင်းပြီး source file တစ်ခုချင်းစီ၏ ရှေ့တွင် #include ဖြင့် ထည့်သွင်းခြင်း ဖြစ်သည်။ Suffix .h သည် header အမည်များအတွက် ရိုးရာ ဖြစ်သည်။ ဥပမာ standard library ၏ functions များကို <stdio.h> ကဲ့သို့ headers များတွင် ကြေညာထားသည်။ ဤအကြောင်းအရာကို Chapter 4 တွင် အသေးစိတ် ဆွေးနွေးပြီး library ကိုယ်တိုင်ကို Chapter 7 နှင့် Appendix B တွင် ဆွေးနွေးသည်။

getline နှင့် copy ၏ အထူးပြု ဗားရှင်းများတွင် arguments များ မရှိသောကြောင့် ယုတ္တိက file ၏ အစတွင် ၎င်းတို့၏ prototypes များသည် getline() နှင့် copy() ဖြစ်သင့်ကြောင်း အကြံပြုလိမ့်မည်။ သို့သော် အဟောင်း C ပရိုဂရမ်များနှင့် လိုက်လျောညီထွေရှိရန်အတွက် standard က ဗလာစာရင်းကို old-style ကြေညာချက်အဖြစ် ယူပြီး argument list စစ်ဆေးမှုအားလုံးကို ပိတ်သည်။ အတိအကျ ဗလာစာရင်းအတွက် void ဟူသော စကားလုံးကို အသုံးပြုရမည်။ ကျွန်ုပ်တို့ ဤအကြောင်းကို Chapter 4 တွင် နောက်ထပ် ဆွေးနွေးမည်။

ဤအပိုင်းတွင် external ကိန်းရှင်များကို ရည်ညွှန်းသောအခါ definition နှင့် declaration ဟူသော စကားလုံးများကို ဂရုတစိုက် အသုံးပြုနေကြောင်း သင် သတိပြုသင့်သည်။ "Definition" သည် ကိန်းရှင်ကို ဖန်တီးသော သို့မဟုတ် သိုလှောင်မှု သတ်မှတ်သော နေရာကို ရည်ညွှန်းသည်။ "Declaration" သည် ကိန်းရှင်၏ သဘောသဘာဝကို ဖော်ပြသော်လည်း သိုလှောင်မှု ခွဲဝေမပေးသော နေရာများကို ရည်ညွှန်းသည်။

ခပ်ဖြင့်၊ မြင်ရသမျှကို extern ကိန်းရှင် ပြုလုပ်လိုသော သဘောထား ရှိသည်၊ အဘယ်ကြောင့်ဆိုသော် ၎င်းက ဆက်သွယ်ရေးများကို ရိုးရှင်းစေသလို ထင်သောကြောင့် - argument lists များ တိုသည်နှင့် သင် လိုချင်သောအခါ ကိန်းရှင်များ အမြဲရှိသည်။ သို့သော် external ကိန်းရှင်များသည် သင် မလိုချင်သောအခါတောင် အမြဲရှိသည်။ External ကိန်းရှင်များအပေါ် အလွန်အကျွံ မှီခိုခြင်းသည် အန္တရာယ်များ ပြည့်နှက်နေသည်၊ အဘယ်ကြောင့်ဆိုသော် ၎င်းက ဒေတာ ချိတ်ဆက်မှုများ အားလုံး ထင်ရှားခြင်း မရှိသော ပရိုဂရမ်များသို့ ဦးတည်စေသောကြောင့် - ကိန်းရှင်များကို မမျှော်လင့်ထားသော နှင့် တမင်မဟုတ်သော နည်းလမ်းများဖြင့်ပင် ပြောင်းလဲနိုင်ပြီး ပရိုဂရမ်ကို ပြုပြင်ရန် ခက်ခဲသည်။ အရှည်ဆုံး-စာကြောင်း ပရိုဂရမ်၏ ဒုတိယဗားရှင်းသည် ပထမထက် နိမ့်ကျသည်၊ တစ်စိတ်တစ်ပိုင်း ဤအကြောင်းရင်းများကြောင့် ဖြစ်ပြီး တစ်စိတ်တစ်ပိုင်း ၎င်းက ကိုင်တွယ်သော ကိန်းရှင်များ၏ အမည်များကို ၎င်းတို့ထဲသို့ ရေးခြင်းဖြင့် အသုံးဝင်သော functions နှစ်ခု၏ ယေဘုယျမှုကို ဖျက်ဆီးသောကြောင့် ဖြစ်သည်။

ဤအချက်တွင် ကျွန်ုပ်တို့သည် C ၏ ရိုးရာ core ဟု ခေါ်နိုင်သည့်အရာကို လွှမ်းခြုံပြီးဖြစ်သည်။ ဤတည်ဆောက်မှု blocks အနည်းငယ်ဖြင့် သိသာထင်ရှားသော အရွယ်အစား၏ အသုံးဝင်သော ပရိုဂရမ်များကို ရေးနိုင်ပြီး သင် အလုံအလောက် ရပ်တန့်ပြီး ထိုသို့ လုပ်ဆောင်လျှင် ကောင်းသော အကြံ ဖြစ်လောက်သည်။ ဤလေ့ကျင့်ခန်းများသည် ဤအခန်း၏ အစောပိုင်းတို့ထက် အနည်းငယ် ပိုကြီးမားသော ရှုပ်ထွေးမှုရှိသော ပရိုဂရမ်များကို အကြံပြုသည်။

လေ့ကျင့်ခန်း 1-20. Input ရှိ tabs များကို နောက်ထပ် tab stop သို့ space လုပ်ရန် သင့်လျော်သော blanks အရေအတွက်ဖြင့် အစားထိုးသော detab ပရိုဂရမ်တစ်ခု ရေးပါ။ Tab stops များ၏ သတ်မှတ်ထားသော set တစ်ခုကို ယူဆပါ၊ ဥပမာ n columns တိုင်း။ n သည် ကိန်းရှင်တစ်ခု သို့မဟုတ် သင်္ကေတ parameter တစ်ခု ဖြစ်သင့်သနည်း။

လေ့ကျင့်ခန်း 1-21. တူညီသော spacing ရရှိရန် blanks များ၏ strings များကို tabs နှင့် blanks များ၏ အနည်းဆုံး အရေအတွက်ဖြင့် အစားထိုးသော entab ပရိုဂရမ်တစ်ခု ရေးပါ။ detab အတွက်ကဲ့သို့ တူညီသော tab stops များကို အသုံးပြုပါ။ tab stop တစ်ခုသို့ ရောက်ရန် tab တစ်ခု သို့မဟုတ် blank တစ်ခုတည်း လုံလောက်သောအခါ မည်သည့်အရာကို ဦးစားပေး သင့်သနည်း။

လေ့ကျင့်ခန်း 1-22. Input ၏ n-th column မတိုင်မီ ဖြစ်ပေါ်သော နောက်ဆုံး non-blank character နောက်တွင် ရှည်လျားသော input lines များကို နှစ်ခု သို့မဟုတ် ထို့ထက်ပိုသော တိုသော စာကြောင်းများအဖြစ် "fold" လုပ်ရန် ပရိုဂရမ်တစ်ခု ရေးပါ။ သင့်ပရိုဂရမ်က အလွန်ရှည်လျားသော စာကြောင်းများဖြင့် ဉာဏ်ရှိသော တစ်ခုခု လုပ်ဆောင်ကြောင်း သေချာစေပါ၊ သတ်မှတ်ထားသော column မတိုင်မီ blanks သို့မဟုတ် tabs များ မရှိပါက။

လေ့ကျင့်ခန်း 1-23. C ပရိုဂရမ်တစ်ခုမှ မှတ်ချက်များအားလုံးကို ဖယ်ရှားရန် ပရိုဂရမ်တစ်ခု ရေးပါ။ Quoted strings များနှင့် character constants များကို မှန်ကန်စွာ ကိုင်တွယ်ရန် မမေ့ပါနှင့်။ C မှတ်ချက်များသည် nest မလုပ်ပါ။

လေ့ကျင့်ခန်း 1-24. မကိုက်ညီသော ကွင်းစကွင်းပိတ်များ၊ ကွင်းထောင့်များနှင့် ကွင်းကောက်များကဲ့သို့ အခြေခံ syntax အမှားများအတွက် C ပရိုဂရမ်တစ်ခုကို စစ်ဆေးရန် ပရိုဂရမ်တစ်ခု ရေးပါ။ Quotes များ single နှင့် double နှစ်ခုစလုံး၊ escape sequences များနှင့် မှတ်ချက်များကို မမေ့ပါနှင့်။ (သင် ၎င်းကို အပြည့်အဝ ယေဘုယျအားဖြင့် လုပ်ပါက ဤပရိုဂရမ်သည် ခက်ခဲသည်။)
