# 2.11 အခြေအနေအရ ဖော်ပြချက်များ (Conditional Expressions)

အောက်ပါ statement များသည်

```
if (a > b)
z = a;
```

else

```
z = b;
```

z တွင် a နှင့် b ၏ အများဆုံးတန်ဖိုးကို တွက်ချက်ပေးပါသည်။ ternary operator ``?:'' ဖြင့် ရေးသားထားသော conditional expression သည် ဤကဲ့သို့သော တည်ဆောက်ပုံများကို ရေးသားရန် အခြားနည်းလမ်းတစ်ခု ဖြစ်ပါသည်။ အောက်ပါ expression တွင်

```
expr1 ? expr2 : expr3
```

expr1 expression ကို ဦးစွာ evaluate လုပ်ပါသည်။ ၎င်းသည် သုည မဟုတ်ပါက (true), expr2 expression ကို evaluate လုပ်ပြီး ၎င်းသည် conditional expression ၏ တန်ဖိုး ဖြစ်ပါသည်။ သို့မဟုတ်ပါက expr3 ကို evaluate လုပ်ပြီး ၎င်းသည် တန်ဖိုး ဖြစ်ပါသည်။ expr2 နှင့် expr3 တို့အနက် တစ်ခုသာ evaluate လုပ်ပါသည်။ ထို့ကြောင့် z ကို a နှင့် b ၏ အများဆုံးတန်ဖိုးအဖြစ် သတ်မှတ်ရန်၊

```
z = (a > b) ? a : b;          /* z = max(a, b) */ 
```

conditional expression သည် expression တစ်ခု ဖြစ်ကြောင်းကို သတိပြုသင့်ပါသည်၊ ထို့ကြောင့် အခြား expression များ အသုံးပြုနိုင်သည့် နေရာတိုင်းတွင် ၎င်းကို အသုံးပြုနိုင်ပါသည်။ အကယ်၍ expr2 နှင့် expr3 တို့သည် မတူညီသော အမျိုးအစားများ ဖြစ်ပါက၊ ရလဒ်၏ အမျိုးအစားကို ဤအခန်းတွင် ယခင်က ဆွေးနွေးခဲ့သော conversion စည်းမျဉ်းများဖြင့် ဆုံးဖြတ်ပါသည်။ ဥပမာအားဖြင့်၊ f သည် float ဖြစ်ပြီး n သည် int ဖြစ်ပါက၊ expression

```
(n > 0) ? f : n
```

သည် n သည် အပြုသဘောဖြစ်သည်ဖြစ်စေ မဖြစ်စေ float အမျိုးအစား ဖြစ်ပါသည်။

conditional expression ၏ ပထမ expression ပတ်လည်တွင် ကွင်းစကွင်းပိတ် လိုအပ်ခြင်းမရှိပါ၊ အဘယ်ကြောင့်ဆိုသော် ?: ၏ precedence သည် အလွန်နိမ့်သောကြောင့်ဖြစ်ပြီး assignment ထက် အနည်းငယ်သာ မြင့်သောကြောင့် ဖြစ်ပါသည်။ သို့သော် ၎င်းတို့သည် expression ၏ condition အပိုင်းကို ပိုမို ရှင်းလင်းစွာ မြင်နိုင်စေသောကြောင့် အကြံပြုထိုက်ပါသည်။

conditional expression သည် များသောအားဖြင့် တိုတောင်းသော code ကို ဖြစ်ပေါ်စေပါသည်။ ဥပမာအားဖြင့်၊ ဈေ loop သည် array ၏ n elements များကို တစ်ကြောင်းလျှင် 10 ခုစီ ပုံနှိပ်ပြသပြီး၊ column တစ်ခုစီကို blank တစ်ခုဖြင့် ခွဲခြားထားကာ၊ တစ်ကြောင်းစီ (နောက်ဆုံးကြောင်းအပါအဝင်) ကို newline ဖြင့် အဆုံးသတ်ပါသည်။

```
for (i = 0; i < n; i++)
printf("%6d%c", a[i], (i%10==9 || i==n-1) ? '\n' : ' ');
```

ဆယ်ခုမြောက် element တိုင်းနောက်တွင် နှင့် n-th element နောက်တွင် newline တစ်ခု ပုံနှိပ်ပါသည်။ အခြား element များအားလုံးသည် blank တစ်ခုဖြင့် နောက်တွေလိုက်ပါသည်။ ဤသည်မှာ လှည့်ကွက်ပြပုံ ရှုပ်ထွေးပုံရပေမယ့်၊ ၎င်းသည် if-else နှင့် ညီမျှသော ပုံစံထက် ပိုမို ကျစ်လစ်ပါသည်။ နောက်ထပ် ကောင်းမွန်သော ဥပမာမှာ

```
printf("You have %d items%s.\n", n, n==1 ? "" : "s"); 
```

Exercise 2-10. upper case စာလုံးများကို lower case အဖြစ် ပြောင်းလဲပေးသော lower function ကို if-else အစား conditional expression ဖြင့် ပြန်လည်ရေးသားပါ။

# 2.12 Precedence နှင့် Order of Evaluation

Table 2.1 သည် ကျွန်ုပ်တို့ မဆွေးနွေးရသေးသော operator များအပါအဝင် operator များအားလုံး၏ precedence နှင့် associativity စည်းမျဉ်းများကို အနှစ်ချုပ်ပြသထားပါသည်။ တူညီသော ကြောင်းတွင်ရှိသော operator များသည် တူညီသော precedence ရှိပါသည်။ အတန်းများသည် precedence လျော့နည်းလာသော အစဉ်အလိုက် ဖြစ်ပါသည်၊ ထို့ကြောင့် ဥပမာအားဖြင့်၊ *, /, နှင့် % တို့သည် တူညီသော precedence ရှိပြီး၊ binary + နှင့် - တို့ထက် မြင့်ပါသည်။ ``operator'' () သည် function call ကို ရည်ညွှန်းပါသည်။ -> နှင့် . operator များကို structures များ၏ member များကို ရယူရန် အသုံးပြုပါသည်။ ၎င်းတို့ကို sizeof (object ၏ အရွယ်အစား) နှင့်အတူ Chapter 6 တွင် လေ့လာရမည်။ Chapter 5 သည် * (pointer မှတစ်ဆင့် indirection) နှင့် & (object ၏ address) တို့ကို ဆွေးနွေးထားပြီး၊ Chapter 3 သည် comma operator ကို ဆွေးနွေးထားပါသည်။

|Operators|Associativity|
|---|---|
|() [] -> .|left to right|
|! ~ ++ -- + - * (type) sizeof|right to left|
|* / %|left to right|
|+ -|left to right|
|<< >>|left to right|
|< <= > >=|left to right|
|== !=|left to right|
|&|left to right|
|^|left to right|
|\||left to right|
|&&|left to right|
|\||left to right|
|?:|right to left|
|= += -= *= /= %= &= ^= \|= <<= >>=|right to left|
|,|left to right|

Unary & +, -, နှင့် * တို့သည် binary ပုံစံများထက် precedence ပိုမြင့်ပါသည်။

Table 2.1: Operator များ၏ Precedence နှင့် Associativity

bitwise operator &, ^, နှင့် | တို့၏ precedence သည် == နှင့် != တို့၏ အောက်တွင် ကျရောက်ကြောင်း သတိပြုပါ။ ဤအချက်က bit-testing expression များဖြစ်သော

```
if ((x & MASK) == 0) ...
```

များကို သင့်လျော်သော ရလဒ်များ ရရှိစေရန် အပြည့်အ၀ ကွင်းစကွင်းပိတ် ထည့်သွင်းရမည်ဟု ဆိုလိုပါသည်။

C သည် အများစုသော ဘာသာစကားများကဲ့သို့ပင် operator ၏ operand များကို မည်သည့်အစီအစဉ်ဖြင့် evaluate လုပ်ရမည်ကို သတ်မှတ်မထားပါ။ (ခြွင်းချက်များမှာ &&, ||, ?:, နှင့် `,' ဖြစ်ပါသည်။) ဥပမာအားဖြင့်၊ statement တစ်ခုတွင်

```
x = f() + g();
```

f ကို g ထက် အရင် evaluate လုပ်နိုင်သည် သို့မဟုတ် ပြောင်းပြန်လည်း ဖြစ်နိုင်ပါသည်။ ထို့ကြောင့် အကယ်၍ f သို့မဟုတ် g တို့သည် တစ်ခုနှင့်တစ်ခု မှီခိုနေသော variable တစ်ခုကို ပြောင်းလဲပါက၊ x သည် evaluation ၏ အစီအစဉ်ပေါ် မူတည်နိုင်ပါသည်။ သတ်မှတ်ထားသော အစီအစဉ်တစ်ခုကို သေချာစေရန် ကြားခံ ရလဒ်များကို ယာယီ variable များတွင် သိမ်းဆည်းနိုင်ပါသည်။

အလားတူပင်၊ function argument များကို evaluate လုပ်သော အစီအစဉ်ကိုလည်း သတ်မှတ်မထားပါ၊ ထို့ကြောင့် statement

```
printf("%d %d\n", ++n, power(2, n)); /* WRONG */ 
```

သည် n ကို power ကို call မလုပ်မီ တိုးမည်လား မတိုးမည်လားပေါ် မူတည်၍ compiler များမတူညီသော မတူညီသော ရလဒ်များ ထွက်ပေါ်နိုင်ပါသည်။ ဖြေရှင်းနည်းမှာ သို့ပေးပေါ့တော့

```
++n;
printf("%d %d\n", n, power(2, n));
```

Function call များ၊ nested assignment statement များ၊ နှင့် increment နှင့် decrement operator များသည် ``side effects'' ကို ဖြစ်ပေါ်စေပါသည် - expression ၏ evaluation ၏ ထုတ်ကုန်ခွဲအဖြစ် variable တစ်ခုအချို့ကို ပြောင်းလဲပါသည်။ side effect များပါဝင်သော မည်သည့် expression တွင်မဆို၊ expression တွင် ပါဝင်သော variable များကို update လုပ်သော အစီအစဉ်ပေါ် သိမ်မွေ့သော မှီခိုမှုများ ရှိနိုင်ပါသည်။ မပျော်ရွှင်ဖွယ် အခြေအနေတစ်ခုကို statement ဖြင့် ဥပမာပြရမည်ဆိုလျှင်

```
a[i] = i++;
```

မေးခွန်းမှာ subscript သည် i ၏ ဟောင်းတန်ဖိုး ဖြစ်သလား သို့မဟုတ် အသစ်တန်ဖိုး ဖြစ်သလား ဖြစ်ပါသည်။ Compiler များသည် ဤအရာကို မတူညီသော နည်းလမ်းများဖြင့် အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်ပြီး၊ ၎င်းတို့၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်ပေါ် မူတည်၍ မတူညီသော အဖြေများ ထုတ်ပေးနိုင်ပါသည်။ standard သည် ရည်ရွယ်ချက်ရှိရှိ အများစုသော ဤကဲ့သို့သော ကိစ္စများကို သတ်မှတ်မထားပါ။ expression တစ်ခုအတွင်း side effect များ (variable များသို့ assignment) မည်သည့်အချိန်တွင် ဖြစ်ပွားသည်ကို compiler ၏ ဆုံးဖြတ်ချက်ပေါ် ချန်ထားထားပါသည်၊ အကောင်းဆုံး အစီအစဉ်သည် machine architecture ပေါ် အလွန်များစွာ မူတည်သောကြောင့် ဖြစ်ပါသည်။ (standard သည် argument များပေါ်ရှိ side effect များအားလုံးသည် function ကို call မလုပ်မီ အကျိုးသက်ရောက်မှု ရှိရမည်ဟု သတ်မှတ်ထားပါသည်၊ သို့သော် ၎င်းသည် အထက်ပါ printf ၏ call တွင် အကူအညီမပေးနိုင်ပါ။)

သိက္ခာတရားမှာ evaluation ၏ အစီအစဉ်ပေါ် မူတည်သော code ရေးသားခြင်းသည် မည်သည့်ဘာသာစကားတွင်မဆို ဆိုးရွားသော programming လေ့ကျင့်ခြင်း ဖြစ်ပါသည်။ သဘာဝအားဖြင့်၊ ရှောင်ကြဉ်ရမည့် အရာများကို သိရန် လိုအပ်ပါသည်၊ သို့သော် အကယ်၍ machine အမျိုးမျိုးတွင် ၎င်းတို့ကို မည်ကဲ့သို့ ပြုလုပ်သည်ကို မသိပါက၊ သင့်အနေဖြင့် သီးခြား implementation တစ်ခု၏ အားသာချက်ကို အသုံးချရန် သွေးဆောင်ခြင်း မခံရပါ။
