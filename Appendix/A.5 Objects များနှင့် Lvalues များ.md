# A.5 Objects များနှင့် Lvalues များ

Object တစ်ခုသည် အမည်ရှိသော သိုလှောင်မှုနယ်ပယ်တစ်ခုဖြစ်ပါသည်၊ lvalue သည် object တစ်ခုကို ရည်ညွှန်းသော expression တစ်ခုဖြစ်ပါသည်။ lvalue expression ၏ ထင်ရှားသော ဥပမာမှာ သင့်လျော်သော အမျိုးအစားနှင့် storage class ရှိသော identifier ဖြစ်ပါသည်။ lvalues များကို ထုတ်ပေးသော အော်ပရေတာများ ရှိပါသည်၊ အကယ်၍ E သည် pointer အမျိုးအစား၏ expression ဖြစ်လျှင်၊ *E သည် E က ညွှန်ပြနေသော object ကို ရည်ညွှန်းသော lvalue expression ဖြစ်ပါသည်။ "lvalue" ဟူသော အမည်သည် assignment expression E1 = E2 မှ ဆင်းသက်လာပြီး၊ ၎င်းတွင် ဘယ်ဘက် operand E1 သည် lvalue expression ဖြစ်ရမည်။ အော်ပရေတာတစ်ခုစီ၏ ဆွေးနွေးချက်သည် ၎င်းက lvalue operands များကို မျှော်လင့်ခြင်း ရှိမရှိနှင့် lvalue တစ်ခုကို ထုတ်ပေးခြင်း ရှိမရှိကို သတ်မှတ်ပါသည်။

# A.6 ပြောင်းလဲမှုများ

အချို့ အော်ပရေတာများသည် ၎င်းတို့၏ operands များပေါ် မူတည်၍၊ operand တစ်ခု၏ တန်ဖိုးကို အမျိုးအစားတစ်ခုမှ အခြားတစ်ခုသို့ ပြောင်းလဲခြင်းကို ဖြစ်စေနိုင်ပါသည်။ ဤအပိုင်းသည် ထိုကဲ့သို့သော ပြောင်းလဲမှုများမှ မျှော်လင့်ရမည့် ရလဒ်ကို ရှင်းပြပါသည်။ Par.6.5 သည် များသောအားဖြင့် သာမန် အော်ပရေတာများက တောင်းဆိုသော ပြောင်းလဲမှုများကို အကျဉ်းချုပ်ပါသည်၊ အော်ပရေတာတစ်ခုစီ၏ ဆွေးနွေးချက်က လိုအပ်သည့်အတိုင်း ဖြည့်စွက်မည်ဖြစ်ပါသည်။

## A.6.1 Integral Promotion

အက္ခရာ၊ short integer သို့မဟုတ် integer bit-field၊ အားလုံးသည် signed သို့မဟုတ် မဟုတ်ခြင်း၊ သို့မဟုတ် enumeration အမျိုးအစား၏ object ကို integer သုံးနိုင်သည့် နေရာတိုင်းတွင် expression တစ်ခုတွင် အသုံးပြုနိုင်ပါသည်။ အကယ်၍ int သည် မူရင်းအမျိုးအစား၏ တန်ဖိုးအားလုံးကို ကိုယ်စားပြုနိုင်လျှင်၊ တန်ဖိုးကို int သို့ ပြောင်းလဲပါသည်၊ မဟုတ်လျှင် တန်ဖိုးကို unsigned int သို့ ပြောင်းလဲပါသည်။ ဤလုပ်ငန်းစဉ်ကို integral promotion ဟုခေါ်ပါသည်။

## A.6.2 Integral Conversions

ကိန်းပြည့်တစ်ခုကို ပေးထားသော unsigned အမျိုးအစားသို့ ပြောင်းလဲရာတွင် ထို ကိန်းပြည့်နှင့် ညီမျှသော (congruent) အသေးဆုံး အနုတ်လက္ခဏာမဟုတ်သော တန်ဖိုးကို ရှာဖွေခြင်းဖြင့် လုပ်ဆောင်ပါသည်၊ modulo သည် unsigned အမျိုးအစားတွင် ကိုယ်စားပြုနိုင်သော အကြီးဆုံးတန်ဖိုးထက် တစ်ခု ပိုပါသည်။ two's complement ကိုယ်စားပြုမှုတွင်၊ အကယ်၍ unsigned အမျိုးအစား၏ bit pattern က ပိုကျဉ်းလျှင် ဤနည်းသည် left-truncation နှင့် ညီမျှပြီး၊ unsigned အမျိုးအစား ပိုကျယ်လျှင် unsigned တန်ဖိုးများကို သုည-ဖြည့်ခြင်းနှင့် signed တန်ဖိုးများကို လက္ခဏာ-တိုးချဲ့ခြင်းနှင့် ညီမျှပါသည်။

ကိန်းပြည့်တစ်ခုကို signed အမျိုးအစားသို့ ပြောင်းလဲသောအခါ၊ အကယ်၍ အသစ်အမျိုးအစားတွင် ကိုယ်စားပြုနိုင်လျှင် တန်ဖိုးသည် မပြောင်းပါ၊ မဟုတ်လျှင် implementation-defined ဖြစ်ပါသည်။

## A.6.3 ကိန်းပြည့်နှင့် Floating

floating အမျိုးအစား၏ တန်ဖိုးတစ်ခုကို integral အမျိုးအစားသို့ ပြောင်းလဲသောအခါ၊ အပိုင်းကိန်းအပိုင်းကို စွန့်ပစ်ပါသည်၊ အကယ်၍ ရလဒ်တန်ဖိုးကို integral အမျိုးအစားတွင် ကိုယ်စားပြု၍မရလျှင်၊ အပြုအမူကို သတ်မှတ်မထားပါ။ အထူးသဖြင့်၊ အနုတ် floating တန်ဖိုးများကို unsigned integral အမျိုးအစားများသို့ ပြောင်းလဲခြင်း၏ ရလဒ်ကို သတ်မှတ်မထားပါ။

integral အမျိုးအစား၏ တန်ဖိုးတစ်ခုကို floating သို့ ပြောင်းလဲသောအခါ၊ တန်ဖိုးသည် ကိုယ်စားပြုနိုင်သော အပိုင်းအခြားတွင် ရှိသော်လည်း အတိအကျ ကိုယ်စားပြု၍မရလျှင်၊ ရလဒ်သည် နောက်ထက်မြင့်သော သို့မဟုတ် နောက်ထက်နိမ့်သော ကိုယ်စားပြုနိုင်သော တန်ဖိုး တစ်ခုခုဖြစ်နိုင်ပါသည်။ အကယ်၍ ရလဒ်သည် အပိုင်းအခြားပြင်ပ ရှိလျှင်၊ အပြုအမူကို သတ်မှတ်မထားပါ။

## A.6.4 Floating အမျိုးအစားများ

တိကျမှုနည်းသော floating တန်ဖိုးတစ်ခုကို ညီမျှသော သို့မဟုတ် ပိုတိကျသော floating အမျိုးအစားသို့ ပြောင်းလဲသောအခါ၊ တန်ဖိုးသည် မပြောင်းပါ။ တိကျမှုပိုသော floating တန်ဖိုးတစ်ခုကို တိကျမှုနည်းသော floating အမျိုးအစားသို့ ပြောင်းလဲသောအခါ၊ တန်ဖိုးသည် ကိုယ်စားပြုနိုင်သော အပိုင်းအခြားအတွင်း ရှိလျှင်၊ ရလဒ်သည် နောက်ထက်မြင့်သော သို့မဟုတ် နောက်ထက်နိမ့်သော ကိုယ်စားပြုနိုင်သော တန်ဖိုး တစ်ခုခုဖြစ်နိုင်ပါသည်။ အကယ်၍ ရလဒ်သည် အပိုင်းအခြားပြင်ပ ရှိလျှင်၊ အပြုအမူကို သတ်မှတ်မထားပါ။

## A.6.5 Arithmetic Conversions

အော်ပရေတာများစွာသည် ပြောင်းလဲမှုများကို ဖြစ်စေပြီး တူညီသောနည်းဖြင့် ရလဒ်အမျိုးအစားများကို ထုတ်ပေးပါသည်။ အကျိုးသက်ရောက်မှုမှာ operands များကို အများနှင့်သုံးသော အမျိုးအစားတစ်ခုသို့ ယူဆောင်လာခြင်းဖြစ်ပြီး၊ ၎င်းသည် ရလဒ်၏ အမျိုးအစားလည်း ဖြစ်ပါသည်။ ဤပုံစံကို သာမန် arithmetic conversions ဟုခေါ်ပါသည်။

• ပထမဦးစွာ၊ အကယ်၍ operand တစ်ခုခု long double ဖြစ်လျှင်၊ အခြားတစ်ခုကို long double သို့ ပြောင်းလဲပါသည်။

• မဟုတ်လျှင်၊ အကယ်၍ operand တစ်ခုခု double ဖြစ်လျှင်၊ အခြားတစ်ခုကို double သို့ ပြောင်းလဲပါသည်။

• မဟုတ်လျှင်၊ အကယ်၍ operand တစ်ခုခု float ဖြစ်လျှင်၊ အခြားတစ်ခုကို float သို့ ပြောင်းလဲပါသည်။

• မဟုတ်လျှင်၊ operands နှစ်ခုစလုံးတွင် integral promotions များကို လုပ်ဆောင်ပါသည်၊ ထို့နောက်၊ အကယ်၍ operand တစ်ခုခု unsigned long int ဖြစ်လျှင်၊ အခြားတစ်ခုကို unsigned long int သို့ ပြောင်းလဲပါသည်။

• မဟုတ်လျှင်၊ အကယ်၍ operand တစ်ခု long int ဖြစ်ပြီး အခြားတစ်ခု unsigned int ဖြစ်လျှင်၊ အကျိုးသက်ရောက်မှုသည် long int က unsigned int ၏ တန်ဖိုးအားလုံးကို ကိုယ်စားပြုနိုင်ခြင်း ရှိမရှိပေါ် မူတည်ပါသည်၊ အကယ်၍ ဖြစ်နိုင်လျှင်၊ unsigned int operand ကို long int သို့ ပြောင်းလဲပါသည်၊ မဖြစ်နိုင်လျှင်၊ နှစ်ခုစလုံးကို unsigned long int သို့ ပြောင်းလဲပါသည်။

• မဟုတ်လျှင်၊ အကယ်၍ operand တစ်ခု long int ဖြစ်လျှင်၊ အခြားတစ်ခုကို long int သို့ ပြောင်းလဲပါသည်။

• မဟုတ်လျှင်၊ အကယ်၍ operand တစ်ခုခု unsigned int ဖြစ်လျှင်၊ အခြားတစ်ခုကို unsigned int သို့ ပြောင်းလဲပါသည်။

• မဟုတ်လျှင်၊ operands နှစ်ခုစလုံးတွင် int အမျိုးအစားရှိပါသည်။

ဤနေရာတွင် ပြောင်းလဲမှုနှစ်ခုရှိပါသည်။ ပထမ၊ float operands များပေါ်တွင် arithmetic ကို double အစား single precision ဖြင့် လုပ်ဆောင်နိုင်ပါသည်၊ ပထမထုတ်ဝေမှုတွင် floating arithmetic အားလုံးသည် double precision ဖြစ်ကြောင်း သတ်မှတ်ခဲ့ပါသည်။ ဒုတိယ၊ တိုတောင်းသော unsigned အမျိုးအစားများကို ပိုကြီးသော signed အမျိုးအစားနှင့် ပေါင်းစပ်သောအခါ၊ unsigned ဂုဏ်သတ္တိကို ရလဒ်အမျိုးအစားသို့ မပြန့်ပွားပါ၊ ပထမထုတ်ဝေမှုတွင်၊ unsigned က အမြဲ လွှမ်းမိုးခဲ့ပါသည်။ စည်းမျဉ်းအသစ်များသည် အနည်းငယ် ပိုရှုပ်ထွေးသော်လည်း၊ unsigned ပမာဏတစ်ခု signed နှင့် တွေ့သောအခါ ဖြစ်နိုင်သော အံ့အားသင့်မှုများကို တစ်စိတ်တစ်ပိုင်း လျှော့ချပေးပါသည်။ unsigned expression တစ်ခုကို အရွယ်အစားတူ signed expression နှင့် နှိုင်းယှဉ်သောအခါ မမျှော်လင့်သော ရလဒ်များ ဆက်လက်ဖြစ်နိုင်ပါသည်။

## A.6.6 Pointers များနှင့် Integers များ

integral အမျိုးအစား၏ expression တစ်ခုကို pointer တစ်ခုသို့ ပေါင်း သို့မဟုတ် နုတ်နိုင်ပါသည်၊ ထိုကဲ့သို့သော ကိစ္စတွင် integral expression ကို ပေါင်းခြင်း အော်ပရေတာ၏ ဆွေးနွေးချက် (Par.A.7.7) တွင် သတ်မှတ်ထားသည့်အတိုင်း ပြောင်းလဲပါသည်။

အမျိုးအစားတူ၊ array တူရှိသော objects နှစ်ခု၏ pointers နှစ်ခုကို နုတ်နိုင်ပါသည်၊ ရလဒ်ကို နုတ်ခြင်း အော်ပရေတာ၏ ဆွေးနွေးချက် (Par.A.7.7) တွင် သတ်မှတ်ထားသည့်အတိုင်း ကိန်းပြည့်သို့ ပြောင်းလဲပါသည်။

တန်ဖိုး 0 ရှိသော integral constant expression သို့မဟုတ် void * အမျိုးအစားသို့ cast လုပ်ထားသော ထိုကဲ့သို့သော expression ကို cast ဖြင့်၊ assignment ဖြင့် သို့မဟုတ် comparison ဖြင့် မည်သည့် အမျိုးအစားမဆို pointer သို့ ပြောင်းလဲနိုင်ပါသည်။ ဤနည်းသည် အမျိုးအစားတူ အခြား null pointer နှင့် ညီမျှသော null pointer တစ်ခုကို ထုတ်လုပ်ပေးပြီး၊ function သို့မဟုတ် object သို့ မည်သည့် pointer နှင့်မျှ မညီပါ။

Pointers များပါသော အခြားအချို့ ပြောင်းလဲမှုများကို ခွင့်ပြုထားပါသည်၊ သို့သော် implementation-defined ရှုထောင့်များ ရှိပါသည်။ ၎င်းတို့ကို တိကျသော type-conversion အော်ပရေတာ သို့မဟုတ် cast (Pars.A.7.5 နှင့် A.8.8) ဖြင့် သတ်မှတ်ရမည်။

Pointer တစ်ခုကို ၎င်းကို ထည့်သွင်းရန် လုံလောက်သော integral အမျိုးအစားသို့ ပြောင်းလဲနိုင်ပါသည်၊ လိုအပ်သော အရွယ်အစားသည် implementation-dependent ဖြစ်ပါသည်။ mapping function လည်း implementation-dependent ဖြစ်ပါသည်။

အမျိုးအစားတစ်ခုသို့ pointer တစ်ခုကို အခြားအမျိုးအစားသို့ pointer တစ်ခုသို့ ပြောင်းလဲနိုင်ပါသည်။ အကယ်၍ subject pointer သည် သိုလှောင်မှုတွင် သင့်လျော်စွာ ချိန်ညှိထားသော (aligned) object တစ်ခုကို ရည်ညွှန်းခြင်း မရှိလျှင်၊ ရလဒ် pointer သည် addressing exceptions များကို ဖြစ်စေနိုင်ပါသည်။ object တစ်ခုသို့ pointer တစ်ခုကို အမျိုးအစားက နည်းသော သို့မဟုတ် ညီမျှစွာ တင်းကျပ်သော storage alignment လိုအပ်သော object သို့ pointer တစ်ခုသို့ ပြောင်းလဲနိုင်ပြီး မပြောင်းလဲဘဲ ပြန်ပြောင်းနိုင်ကြောင်း အာမခံပါသည်၊ "alignment" ၏ အယူအဆသည် implementation-dependent ဖြစ်သော်လည်း၊ char အမျိုးအစားများ၏ objects များတွင် အလွတ်ဆုံး တင်းကျပ်သော alignment လိုအပ်ချက်များ ရှိပါသည်။ Par.A.6.8 တွင် ဖော်ပြထားသည့်အတိုင်း၊ pointer ကို void * အမျိုးအစားသို့လည်း မပြောင်းလဲဘဲ ပြန်ပြောင်းနိုင်ပါသည်။

Pointer တစ်ခုကို pointer က ရည်ညွှန်းသော object အမျိုးအစား၏ qualifiers (Pars.A.4.4, A.8.2) များကို ပေါင်းထည့်ခြင်း သို့မဟုတ် ဖယ်ရှားခြင်းမှလွဲ၍ အမျိုးအစားတူ အခြား pointer တစ်ခုသို့ ပြောင်းလဲနိုင်ပါသည်။ အကယ်၍ qualifiers များကို ပေါင်းထည့်လျှင်၊ pointer အသစ်သည် qualifiers အသစ်များက ညွှန်ပြသော ကန့်သတ်ချက်များမှလွဲ၍ ဟောင်းနှင့် ညီမျှပါသည်။ အကယ်၍ qualifiers များကို ဖယ်ရှားလျှင်၊ အောက်ခံ object ပေါ်တွင် လုပ်ဆောင်မှုများသည် ၎င်း၏ အမှန်တကယ် ကြေညာချက်ရှိ qualifiers များအောက်တွင် ဆက်လက်ရှိနေပါသည်။

နောက်ဆုံးအနေဖြင့်၊ function တစ်ခုသို့ pointer တစ်ခုကို အခြား function အမျိုးအစားသို့ pointer တစ်ခုသို့ ပြောင်းလဲနိုင်ပါသည်။ ပြောင်းလဲထားသော pointer က သတ်မှတ်ထားသော function ကို ခေါ်ဆိုခြင်းသည် implementation-dependent ဖြစ်ပါသည်၊ သို့သော်၊ အကယ်၍ ပြောင်းလဲထားသော pointer ကို ၎င်း၏ မူရင်းအမျိုးအစားသို့ ပြန်ပြောင်းလျှင်၊ ရလဒ်သည် မူရင်း pointer နှင့် ထပ်တူဖြစ်ပါသည်။

## A.6.7 Void

void object ၏ (မရှိသော) တန်ဖိုးကို မည်သည့်နည်းဖြင့်မျှ အသုံးပြု၍မရပါ၊ မည်သည့် non-void အမျိုးအစားသို့မဆို တိကျသော သို့မဟုတ် သွယ်ဝိုက်သော ပြောင်းလဲမှုကိုလည်း အသုံးပြု၍မရပါ။ void expression တစ်ခုသည် မရှိသော တန်ဖိုးကို ညွှန်ပြသောကြောင့်၊ ထိုကဲ့သို့သော expression ကို တန်ဖိုး မလိုအပ်သည့်နေရာတွင်သာ အသုံးပြုနိုင်ပါသည်၊ ဥပမာ expression statement (Par.A.9.2) အဖြစ် သို့မဟုတ် comma အော်ပရေတာ (Par.A.7.18) ၏ ဘယ်ဘက် operand အဖြစ်။

Expression တစ်ခုကို cast ဖြင့် void အမျိုးအစားသို့ ပြောင်းလဲနိုင်ပါသည်။ ဥပမာ၊ void cast သည် expression statement အဖြစ် အသုံးပြုထားသော function call ၏ တန်ဖိုးကို စွန့်ပစ်ခြင်းကို မှတ်တမ်းတင်ပါသည်။

void သည် ဤစာအုပ်၏ ပထမထုတ်ဝေမှုတွင် မပေါ်ခဲ့ပါ၊ သို့သော် နောက်ပိုင်းတွင် အဖြစ်များလာပါသည်။

## A.6.8 Void သို့ Pointers များ

မည်သည့် object သို့ pointer မဆို void * အမျိုးအစားသို့ အချက်အလက် ဆုံးရှုံးခြင်းမရှိဘဲ ပြောင်းလဲနိုင်ပါသည်။ အကယ်၍ ရလဒ်ကို မူရင်း pointer အမျိုးအစားသို့ ပြန်ပြောင်းလျှင်၊ မူရင်း pointer ကို ပြန်ရရှိပါသည်။ Par.A.6.6 တွင် ဆွေးနွေးထားသော pointer-to-pointer conversions များနှင့် မတူဘဲ (ယေဘုယျအားဖြင့် တိကျသော cast လိုအပ်ပါသည်)၊ pointers များကို void * အမျိုးအစား၏ pointers များမှ သတ်မှတ်နိုင်ပြီး၊ ၎င်းတို့နှင့် နှိုင်းယှဉ်နိုင်ပါသည်။

void * pointers များ၏ ဤအဓိပ္ပါယ်ဖွင့်ဆိုချက်သည် အသစ်ဖြစ်ပါသည်။ ယခင်က၊ char * pointers များက generic pointer ၏ အခန်းကဏ္ဍကို ပါဝင်ခဲ့ပါသည်။ ANSI စံချိန်စံညွှန်းသည် assignments နှင့် relationals များတွင် void * pointers များကို object pointers များနှင့် တွေ့ဆုံခြင်းကို သီးခြားအတည်ပြုပြီး၊ အခြား pointer ရောနှောမှုများအတွက် တိကျသော casts များကို လိုအပ်ပါသည်။
